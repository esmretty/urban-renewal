## 觀察 / 待確認

### 1. 「道路用地」資訊未進 DB（2 筆 mismatch 的共同原因）

**現象**：兩筆 mismatch（#1 中華路60巷16號 / #2 華中街47巷2號）信義原網頁都標「**第四種住宅區; 道路用地**」，但 DB 只存「第四種住宅區」。

**根因（已查清）**：信義 scraper **完全沒有抓 zoning 欄位**（`scraper_sinyi.py` grep zoning 0 結果）。DB 的 zoning 是後續 pipeline 走 `arcgis_newtaipei`（新北市都市計畫 ArcGIS）用 lat/lng **點查詢**得來：
```
zoning_source = 'arcgis_newtaipei'
zoning_source_url = 'https://urban.planning.ntpc.gov.tw/NtpcURInfo/'
```

信義網頁標的是「整塊地」的分區（橫跨兩個分區）；ArcGIS 點查詢只回該座標**單一點**所在分區 → 看不到「另一部分是道路用地」這個資訊。**不是 scraper bug，是方法論差異**。

**為什麼重要**：
- 「道路用地」= 該地塊有部分被計畫道路徵收，**不能蓋**
- 都更分回試算用整塊土地坪數 × 容積率（300%）會**虛報效益** — 實際可建坪數要扣掉道路用地那塊
- 用戶若依 DB 顯示去出價，可能誤判「這塊地可以蓋很多」最後吃悶虧

**建議行動（待用戶決定）**：
- 短期：信義 scraper 加抓「使用分區」HTML 字串，存進 `zoning_sinyi_raw` 之類的補欄位（不覆蓋 ArcGIS 主欄）。前端若 `zoning_sinyi_raw` 含「道路用地」「公共設施保留地」「停車場用地」等 → 顯示警示
- 中期：永慶 scraper 也加同樣抽取（永慶 HTML 也有 `class="item-detail">XXX</div>` 直接抓得到）
- 長期：思考 ArcGIS 是否能改用「polygon 查詢」拿到整塊地多分區（但成本/邊界判定複雜）

### 2. `zoning` 跟 `zoning_original` 兩欄位永遠相同

21 筆裡兩個欄位內容**完全一致**（全是「住宅區」「第四種住宅區」「商業區」這類人類可讀字串）。

對比 591 source：591 經 Taipei GeoServer 拿到的是「LZ3」「LZ4」這類代碼 → 寫進 `zoning`，原文（「第三種住宅區」）寫進 `zoning_original`。永慶/信義因為原網頁就給人類可讀字串，沒有 normalize 過程 → 兩欄位重複。

**不是 bug，但**：可以考慮永慶/信義 source 直接把代碼也補上（例：「住宅區」→ R）讓下游試算邏輯統一處理。或者文件化「zoning_original = 顯示用、zoning = 機器用」的 contract，讓兩個 source 各自填對。

### 3. 永慶「住宅區」沒分種類

永慶 11 筆裡 10 筆是「住宅區」(沒分一二三四種)、1 筆「商業區」。對比信義：新店區 4 筆全是「第四種住宅區」。

**疑問**：永慶網頁是否真的只標「住宅區」三個字？還是 scraper 用太鬆 regex 把「第三種住宅區」吞成「住宅區」？

從 fetch_yongqing 抓到的字串看（HTML `class="item-detail">住宅區</div>`）— 確實是網頁原文就只寫「住宅區」，沒分種類。**這是永慶網站的資料粒度問題，不是 scraper bug**。但這代表新北市永慶物件**無法用「第幾種」精準判 FAR**，目前 [config.py](config.py) 的 contains-match（含「住」→ 300%）已經正確處理這個 case。

### 4. 一筆有意思：永慶「商業區」(yongqing_7337654, 板橋公園街)

板橋商業區應該是 460%（[config.py](config.py) 板橋特例），但目前所有永慶物件 zoning_original 都跟 zoning 相同。要確認 [config.py:lookup_far](config.py) 對「商業區」字串能正確 dispatch 到板橋 460。從 contains-match 邏輯（含「商」→ 板橋 460 / 其他 440）看是對的，**但建議實測這筆的 multiple 計算**。
